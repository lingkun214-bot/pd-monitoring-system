# 全屏导航交互修复设计

## 背景

局放在线监测系统通过右上角“⛶”按钮调用 Fullscreen API。初始实现把
`document.documentElement` 作为全屏目标，且没有监听全屏状态变化或失败事件。
第一轮修复已将原生全屏目标改为 `#appShell`，该方案在 HTTP 模式下通过了
浏览器验收。

用户的实际运行方式是通过 `file://` 直接打开 `index.html`。第二轮修复为
该协议增加了 `#appShell.is-viewport-fullscreen` CSS 回退，但应用外壳本身
已经是 `100vw × 100vh`，添加固定定位只能铺满现有内容视口，不能隐藏浏览器
框架或产生可见放大。按钮状态虽然变化，但不满足用户对“全屏”的定义。

最终方案改为让根文档进入原生全屏，并保留已经补齐的状态监听、异常处理和
统一导航逻辑。根文档包含完整应用外壳，避免只把某个应用子树放入顶层全屏
导致的点击路由差异。

现有页面切换监听器在普通窗口和扩大视口下均能正确切换上述页面，导航按钮
中心点也未被页面元素覆盖。因此修复范围限定为全屏目标与全屏生命周期管理，
不调整页面切换业务逻辑。

## 目标

- 点击右上角“⛶”后，让完整文档进入浏览器原生全屏并产生可见放大。
- 全屏状态下八个顶部导航按钮均保持可点击，尤其覆盖用户报告的后四个页面。
- 进入和退出全屏时，按钮的可访问名称、提示和图标状态保持一致。
- 全屏失败时给出可见反馈，不产生未处理的 Promise 拒绝。
- 全屏切换后重新计算布局并重绘图表。
- 在 `file://` 与 HTTP/HTTPS 模式下，第二次点击都必须退出原生全屏并恢复
  进入前的浏览器内容尺寸。

## 非目标

- 不再使用 CSS 视口铺满模拟浏览器原生全屏。
- 不调整各业务页面的数据、表格、设备树或操作流程。
- 不修改顶部导航的视觉风格和信息架构。
- 不尝试移动、最大化或控制操作系统窗口；仅使用标准 Fullscreen API。

## 设计

### 统一原生全屏目标

使用 `document.documentElement` 作为唯一原生全屏目标，不再按协议分支：

```text
点击“⛶”
  → document.documentElement.requestFullscreen()
  → fullscreenchange
  → 同步按钮状态并重绘

再次点击“⛶”
  → document.exitFullscreen()
  → fullscreenchange
  → 恢复按钮状态并重绘
```

根文档包含登录视图和完整应用外壳；登录后只有应用外壳可见，因此顶部导航、
主内容区、状态栏和全屏按钮都处于同一个原生全屏子树内。此方案既能产生真实
放大，也避免全屏按钮或导航节点落在全屏根节点之外。

### 状态模型

状态判断由一个统一函数提供：

```text
fullscreenTarget = document.documentElement
active = document.fullscreenElement === fullscreenTarget
```

切换函数必须依据当前实际状态执行进入或退出，不保留
`is-viewport-fullscreen` 或协议解析分支。

### 状态同步

新增一个只负责呈现状态的同步函数，在以下时机调用：

- 初始化完成后；
- `fullscreenchange` 事件触发后；
- 全屏请求失败后。

同步内容包括：

- `aria-label`：`进入全屏` / `退出全屏`；
- `title`：`全屏` / `退出全屏`；
- 按钮文本保持现有“⛶”，全屏时可切换为清晰的退出符号；
- 在下一帧调用现有 `drawAll()`，使 Canvas 图表适应新尺寸。

### 错误处理

全屏按钮处理函数使用 `try/catch`：

- 浏览器不支持 `requestFullscreen` 时提示“当前浏览器不支持全屏模式”；
- 请求或退出失败时提示“全屏切换失败，请重试”；
- 错误发生后调用状态同步函数，确保按钮文案与真实状态一致。

### 样式约束

为 `:root:fullscreen` 和其内部 `#appShell` 设置：

- `width: 100vw`；
- `height: 100vh`；
- 保持现有背景色和 `overflow: hidden`；
- 顶部导航继续使用现有 `z-index`，不通过额外超高层级掩盖问题。

删除 `#appShell.is-viewport-fullscreen` 的固定定位规则，防止按钮状态与真实
浏览器全屏状态再次分离。

## 测试

### 自动化回归

结构测试应验证：

1. 全屏请求目标为 `document.documentElement`。
2. 注册了 `fullscreenchange` 监听器。
3. 全屏按钮具有异常捕获与状态同步路径。
4. 根文档全屏时 `#appShell` 具有明确尺寸和溢出规则。
5. 现有八个导航按钮与统一页面切换监听器保持不变。
6. `file://` 与 HTTP/HTTPS 使用同一个原生全屏路径。
7. 代码和样式中不再存在 `is-viewport-fullscreen`。
8. 第二次切换调用 `document.exitFullscreen()` 并恢复“进入全屏”按钮状态。

### 浏览器验收

分别在 HTTP 与 `file://` 模式下，使用右上角“⛶”进入全屏后，依次点击：

1. 数据处理；
2. 报警管理；
3. 设备管理；
4. 系统设置。

每次点击后应满足：

- 对应导航按钮进入激活状态；
- 对应页面可见；
- 顶部导航仍可继续操作；
- 页面无横向溢出；
- 退出全屏后仍可正常导航。

文件模式还必须由用户在实际 `file://` 标签页验证：

- 点击后浏览器框架隐藏，内容产生可见放大；
- 全屏后八个主导航按钮仍可切换页面；
- 第二次点击后退出原生全屏；
- 按钮恢复 `aria-label="进入全屏"`，页面尺寸与导航布局恢复。

## 兼容与回退

任一协议无法使用原生 Fullscreen API 时保留普通页面布局，并通过提示告知
用户。请求被拒绝或退出失败时统一捕获异常、同步按钮状态，不再降级为没有
可见放大效果的 CSS 假全屏。
