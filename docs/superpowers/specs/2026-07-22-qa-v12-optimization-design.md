# 回归测试 V1.2 问题优化设计规格

## 1. 背景与目标

本规格依据《局部放电在线监测系统_整改后回归测试报告及优化建议_V1.2》处理 NEW-001 至 NEW-006，并把 NEW-007 明确保留为生产能力边界。本轮目标是在不扩大静态原型架构的前提下，关闭历史查询、时间语义、数据新鲜度、异步反馈、诊断日期和硬件配置职责问题，使页面结果、导出内容和用户反馈保持一致。

## 2. 范围

### 2.1 本轮实现

- NEW-001：历史时间范围参与查询、回放和导出。
- NEW-002：采集数据时间与配置、操作时间分离。
- NEW-003：系统日志以 UTC 存储，以 Asia/Shanghai（UTC+8）展示和导出。
- NEW-004：按钮操作使用统一结果合同，失败不进入成功状态。
- NEW-005：诊断报告默认日期使用上海时区当天日期。
- NEW-006：设备管理作为唯一硬件编辑入口；系统设置仅提供只读总览和跳转。

### 2.2 生产能力边界

NEW-007 不在静态原型中模拟为真实安全能力。页面需明确说明：演示用户不控制固定 `admin` 登录；真实认证、会话失效、最小权限、接口鉴权和不可篡改审计必须由服务端与安全验收提供。

## 3. 方案选择

采用“规则集中化”方案：时间与操作规则放入 `core.js` 纯函数，`app.js` 负责状态编排和 DOM 联动，`index.html` 与 `styles.css` 仅承载必要控件和视觉反馈。保留现有单页原型结构，不进行大规模文件拆分。

## 4. 历史时间查询

### 4.1 控件与默认值

- 将当前时间范围文本输入拆为开始时间和结束时间两个 `datetime-local` 控件，并提供稳定 ID。
- 页面初始化时，以 Asia/Shanghai 为业务时区，默认范围为演示数据覆盖区间，确保首次进入能够看到记录。
- 提供“最近24小时”“最近7天”“最近30天”快捷入口。
- 最大允许跨度为 90 天。

### 4.2 校验规则

- 开始和结束时间必须同时存在并能解析。
- 开始时间不得晚于结束时间。
- 时间跨度不得超过 90 天。
- 输入按 Asia/Shanghai 解释，再转换为可比较时间戳。
- 校验失败时保留原输入，显示可访问错误消息，不改变现有结果和回放状态。

### 4.3 查询与导出一致性

- 时间范围、机组/通道、告警级别共同传入统一筛选函数。
- 查询结果、结果数量、列表、回放选中项和导出数据使用同一个筛选结果集合。
- CSV、JSON 和波形导出回显开始时间、结束时间、业务时区、机组/通道和级别。
- 无数据时列表展示明确空状态，禁用依赖选中记录的回放，并保留导出条件说明。
- 重置恢复默认范围、全部机组和全部级别，并重新生成列表。

## 5. 时间语义与数据新鲜度

### 5.1 时间类型

- `lastDataUpdatedAt` 只表示采集数据抵达时间。
- 配置保存、导出、筛选、报告生成和日志追加属于操作时间，不得修改采集时间。
- 需要展示保存时间的面板使用独立配置时间反馈，不复用状态栏采集时间。

### 5.2 状态栏

- 文案调整为“采集数据：刚刚 / N秒前 / N分钟前”。
- 过期判断只依据采集数据时间。
- 原型内模拟新采集数据的定时路径可以更新采集时间；普通按钮操作不得更新。

### 5.3 系统日志

- 日志记录继续保存 ISO 8601 UTC 字符串。
- 页面列表、详情和 CSV 使用同一 Asia/Shanghai 格式化函数。
- 展示和导出列名明确包含 `UTC+8`。
- 非法时间显示“时间无效”，不得使日志渲染失败。

## 6. 异步操作合同

### 6.1 统一结果

按钮操作统一返回：

```js
{ ok: true, data }
{ ok: false, error }
```

兼容同步值、Promise 和显式抛错；页面层将返回值归一化后再决定状态。

### 6.2 状态转换

- 开始操作：进入 loading，禁用当前按钮并显示忙碌文案。
- `ok: true`：进入 ready，恢复按钮；由具体业务显示成功提示。
- `ok: false` 或抛错：进入 error，保留输入和现有结果，展示错误与重试入口。
- 任何普通业务操作均不得调用采集时间更新函数。
- 滤波校验失败、诊断未完成、导出未生成等情况必须返回失败结果，不能显示成功 Toast。

## 7. 诊断报告日期

- HTML 不再使用固定日期值。
- 初始化时通过上海时区日期函数填入当天 `YYYY-MM-DD`。
- 用户修改后保持用户值，预览、下载和打印均读取同一日期。
- 报告数据可记录日期来源为“系统默认”或“用户修改”，供演示追踪。

## 8. 硬件配置职责

- 设备管理保留设备行选择、字段编辑、校验、保存和未保存切换保护，是唯一维护入口。
- 系统设置 / 硬件配置改为只读总览，数据与设备管理当前设备集合保持一致。
- 总览显示通道 ID、机组、名称、类型和状态，不出现可编辑控件。
- 增加“前往设备管理”按钮，点击后切换到设备管理页并聚焦设备表格或当前通道。
- 页面文案明确“配置修改请前往设备管理”，避免双入口口径冲突。

## 9. 权限与审计边界

- 用户管理继续作为资料和启停状态的前端演示。
- 边界提示必须明确：停用演示用户不会改变固定 `admin` 登录。
- 系统日志明确为可修改的本地演示记录，不能作为生产审计证据。
- README 与相关页面说明生产验收需要服务端认证、RBAC、会话控制和不可篡改审计。

## 10. 文件职责

- `core.js`：时间范围解析和校验、历史组合筛选、上海时区格式化、操作结果归一化。
- `app.js`：历史查询状态、快捷范围、导出条件、采集时间状态、按钮操作编排、报告日期初始化、硬件总览跳转。
- `index.html`：时间控件、快捷按钮、错误/空状态、UTC+8 标签、只读硬件总览和跳转按钮。
- `styles.css`：新增控件与提示的响应式布局，保持现有深蓝工业监测视觉。
- `tests/core.test.js`：纯函数边界、时区、组合筛选与操作结果合同。
- `tests/structure.test.js`：DOM 稳定节点、页面联动路径和生产边界文案。

## 11. 错误处理

- 时间缺失、解析失败、倒序、超过 90 天分别返回明确错误。
- 无历史结果不是系统异常，显示空状态并禁止无数据回放。
- 日志时间非法时局部降级，不阻断整张表。
- 操作失败不清空用户输入、不刷新采集时间、不显示成功反馈。
- 跳转目标不存在时显示通用错误反馈，不静默失败。

## 12. 测试与验收

### 12.1 自动化

- 时间范围包含边界值，排除范围外记录。
- 上海时区输入转换、UTC 日志转 UTC+8、非法时间降级。
- 时间、机组/通道、级别组合筛选。
- 查询筛选条件与导出载荷一致。
- 90 天跨度上限、倒序、缺失值和无结果。
- `{ ok: false }` 与抛错均进入失败路径，`{ ok: true }` 才成功。
- 普通操作路径不调用采集时间更新。
- 报告日期初始化为上海当天。
- 系统硬件配置为只读且存在设备管理跳转。

### 12.2 浏览器回归

- 历史快捷范围、手动范围、组合筛选、无数据、重置、回放和三类导出。
- 失败操作保留输入、展示错误和重试，不显示假成功。
- 新增日志在页面和 CSV 中显示正确 UTC+8 时间。
- 诊断报告默认日期正确，修改后预览和导出一致。
- 系统设置跳转设备管理并聚焦目标。
- 1366×768 与 1920×1080 下无意外横向溢出。

## 13. 准出条件

- NEW-001 至 NEW-006 的自动化与浏览器场景全部通过。
- 历史查询条件、页面列表、回放和导出完全一致。
- 采集时间不被普通业务操作刷新。
- 失败操作不显示成功状态。
- 系统设置不再形成第二个硬件编辑入口。
- NEW-007 的原型与生产边界在页面和 README 中可见且一致。
